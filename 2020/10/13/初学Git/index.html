初学Git

发表于 2020-10-13 更新于 2020-10-18 分类于 Git 阅读次数：
  • 仓库的创建以及代码文件的创建

  • Git的命令

    • 直接在源文件之中进行编写

    • 注意在git log命令之后，有可以标识每次修改之后的文件状态的指针，（这个很重要），有了这个指针之后，就可以利用

      回退文件到你想要回退的版本之中。

      注意！！，git add仅仅是将文件添加到缓存区(stage)，必须要使用git commit 才能将文件添加到分支里面去。

    • git status 是查看此时的工作区情况，告诉此时工作区有哪些变化，eg：有哪些文件是新添加进来的，有哪些文件是发生了修改的，同时这些文件都是在工作区，还没有进入到缓存区，当所有的文件都通过git commit -m “message”提交到了分支之中时，通过log就可以查看添加的文件的指针，方便在后面对需要重新返回的文件进行reset操作。
    • Git管理的是修改，所以每次修改之后，要用add把修改添加到缓存区，添加到缓存区之后，才能利用commit添加到仓库里面，简言之，commit就是你最近一次add或者commit命令之后的缓存区。
    • 注意，每一次你需要在将文件从版本库里面添加或者删除的时候，都需要有commit命令一起使用
    • 同时，只要add之后，都会添加到版本库里面，只要在版本库里面有的文件，如果误删，都可以用git checkout — 进行恢复，但是恢复的时候，是没有了最近的修改的。

  • 工作区，缓存区

    • 工作区：edit了你的文件之后还没add之前
    • add之后，你的文件就保存在了缓存区，等待进一步的commit

    • Git管理的是你的修改，你的每一次的修改，都会保存在记录里面，便于你自己回退

      • 场景1：当你改乱了工作区某个文件的内容，想直接丢弃工作区的修改时，用命令git checkout — file。

      • 场景2：当你不但改乱了工作区某个文件的内容，还添加到了暂存区时，想丢弃修改，分两步，第一步用命令git reset HEAD ，就回到了场景1，第二步按场景1操作。

    * 场景3：已经提交了不合适的修改到版本库时，想要撤销本次提交，参考版本回退一节，不过前提是没有推送到远程库。

* 分支管理

由于分支的创建和合并很快，所以很推荐使用分支，避免add和commit的次数过多。
    * 解决冲突：git status告诉冲突，然后，自己手动解决冲突......(啊这。。)
master分支应该是很**稳定**的，所以在工作的时候，避免直接在master分支进行操作，在其他的分支上进行操作，最后再合并，这样会使得分支的管理更加稳定，同时利于多人协作。

**善用分支**：每次只要不是很大程度上的改变，都新开一个分支进行测试，最后再把所有完成了的东西全部添加到master分支上，只有这样，才能使得master分支更加稳定。
